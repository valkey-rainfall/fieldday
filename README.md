# fieldday

Generate presentation-quality SVG diagrams of C struct memory layouts —
fields drawn proportionally to their size, padding holes made visible,
bitfields included, cache-line boundaries marked, colors themeable for
blog posts and slide decks.

**Try it in your browser (no setup):** https://valkey-rainfall.github.io/fieldday/

```
fieldday struct.c -o diagram.svg
```

Two implementations, one behavior:

- **Python CLI** (this repo's `src/`): the reference. Layout data (offsets,
  sizes, padding, bitfield placement) is computed by **compiling and running
  a probe program with your system C compiler** — not by reimplementing ABI
  rules — so the diagram shows exactly what the compiler does.
- **Browser app** (`docs/`, served by GitHub Pages): a dependency-free
  JavaScript port. Its layout engine is cross-validated in CI against golden
  vectors generated by the compiler probe, and its renderer is verified
  byte-identical to the Python renderer.

Layouts reflect x86-64 SysV. Unsupported constructs produce an explicit
error instead of a guess.

## Install (CLI)

```
pip install -e .          # installs the `fieldday` command
```

Python >= 3.9, `pycparser`, and any C compiler (`cc`/`gcc`/`clang`) on PATH.

## Usage

Input is a C snippet containing one or more struct definitions:

```c
/* client.c */
struct client {
    uint64_t id;
    int fd;
    uint8_t resp;
    sds name;              /* unknown types: see stubs below */
    unsigned flags : 12;   /* bitfields supported */
    void *conn;
};
```

```
fieldday client.c                     # writes client.svg next to the input
fieldday client.c -o out.svg          # explicit output path
cat client.c | fieldday - -o out.svg  # read from stdin
fieldday client.c --struct client     # pick one struct from a multi-struct snippet
```

Presentation flags:

```
fieldday client.c --theme dark --transparent --title "struct client (Valkey 9.1)"
fieldday client.c --px-per-byte 20    # horizontal scale; match across figures
fieldday client.c --responsive        # fluid-width SVG (mobile-friendly inlining)
fieldday client.c --no-ruler          # omit byte rulers
fieldday client.c --cache-line 0      # disable cache-line rules (default 64)
fieldday client.c --padding-callout   # add "N of M bytes are padding" note
fieldday client.c --css mystyle.css   # extra CSS appended into the SVG
```

Run `fieldday --help` for the full list.

## What the diagram shows

- One box per field, width proportional to size; byte ruler underneath
- Padding as hatched boxes — including sub-byte bitfield-unit padding
- Bitfields as bit-precise boxes labeled `name:width`
- Pointer fields prefixed `*`; flexible array members as a dashed end marker
- Arrays and embedded structs show faint internal boundaries (elements /
  nested fields) without label clutter
- **Cache-line boundaries** (default 64B) as bold dashed rules cutting
  through the diagram, so a field straddling a line is visible at a glance
- Labels that don't fit their box are hoisted above the bar with
  collision-free right-angle leader lines

## Unknown types (stubs)

The snippet doesn't need headers. Unknown type names resolve in order:

1. **Builtin stubs** — fixed-width ints (`uint32_t`, ...), `size_t`, `sds`,
   `time_t`, `bool`, and other common names are known already.
2. **Same-snippet structs** — structs defined earlier (separately, via
   typedef, or inline) can be used as field types.
3. **Stub directives** — declare anything else with a comment line:

```c
//@ stub robj 16 8        <- name, size in bytes, optional alignment
struct entry { robj *obj; };
```

## Annotations (hand-tweak workflow)

For diagrams that need manual edits — companion allocations, savings notes,
custom titles, relabeled or hidden members — export the layout, edit it,
re-render:

```
fieldday client.c --emit-json > layout.json
$EDITOR layout.json
fieldday --from-json layout.json -o final.svg
```

Struct-level annotation keys:

```jsonc
{
  "title": "Valkey 9.1: element embedded in the node",   // hand-written title
  "note": "51 bytes in 1 allocation — 15% smaller",       // green note line
  "extras": [                                             // companion allocations
    {"label": "element sds (16B + hdr)", "bytes": 19, "kind": "embedded"},
    {"label": "out-of-line value", "bytes": 24, "kind": "separate"}
  ],
  "relabel": {"val_ptr": "*ptr (optional)", "lru": ""}    // rename; "" hides
}
```

- `embedded` extras continue the allocation (translucent dashed boxes; the
  byte ruler continues across them)
- `separate` extras start a new allocation after a `+` (own 0-based ruler);
  embedded extras listed after a separate one belong to that allocation

The web app exposes all of this as per-struct form fields (switch structs
with the picker) and downloads the same JSON.

## Theming

Builtin themes: `valkey` (light, default — matches the valkey.io blog) and
`dark`. Custom theme = JSON file with any subset of these keys:

```json
{ "background": "#ffffff", "text": "#002a3a", "muted": "#667788",
  "field-fill": "#6983ff", "field-text": "#ffffff", "field-border": "#30176e",
  "padding-fill": "#f5f7f7", "padding-stroke": "#b9c2cc",
  "highlight": "#1e8e3e", "font": "'Fira Mono', monospace" }
```

Every color also resolves through a CSS custom property named identically
(`--fd-field-fill`, `--fd-highlight`, ...). **Inline the SVG into a page and
define `--fd-*` variables in the page's CSS** — the diagram follows the page
theme, including dark/light switches, with no re-render. `--transparent`
omits the background so the page shows through. Elements carry readable
classes (`fd-field-box`, `fd-padding-label`, `fd-cache-line`, ...) for
targeted styling via `--css` / the web Custom CSS panel.

## Examples and gallery

`examples/` holds small feature-demo snippets. `gallery/` holds the
annotated layout JSONs behind published diagrams, one directory per
deliverable — these are the sources; rendered SVGs are build artifacts
that live at their destination, not in this repo. Some gallery files
double as CI fixtures for the renderer parity check.

## Development

```
python3 -m venv .venv && .venv/bin/pip install -e .[dev]
.venv/bin/pytest                    # layout correctness vs the compiler; renderer; CLI
node tests/js/run_vectors.mjs       # JS engine vs compiler-generated golden vectors
node tests/js/run_geometry.mjs      # SVG geometric invariants
tests/js/run_parity.sh              # Python and JS renderers byte-identical
.venv/bin/python tools/gen_vectors.py   # regenerate golden vectors (CI checks drift)
```

The Python implementation is the reference: new layout semantics land there
first, flow into `tests/vectors.json` via the generator, and the JS engine
must reproduce every vector exactly.
